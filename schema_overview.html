Banco de Dados

O banco de dados da Plataforma Tenda é projetado para armazenar e gerenciar todas as informações necessárias para o funcionamento do sistema. Abaixo, você encontrará uma explicação detalhada de como os dados são organizados e estruturados.

Estrutura do Banco de Dados

A plataforma utiliza várias tabelas interligadas para armazenar diferentes tipos de informações. Aqui está uma visão geral das principais tabelas:

  • cidades: Armazena informações sobre as cidades, conectadas aos estados.
  • famílias: Registros de famílias, vinculados às tendas.
  • missas: Eventos de missa, vinculados às paróquias.
  • paróquias: Informações sobre paróquias, conectadas às cidades e estados.
  • padres: Registros de padres, vinculados aos perfis de usuários e paróquias.
  • funções: Funções de usuário (devAdmin, Fundador, Guardião, etc.).
  • funções_serviço: Funções de equipe/serviço (Líder, Participante, etc.).
  • etapas: Etapas de eventos ou tendas.
  • estados: Informações sobre estados.
  • variáveis_sistema: Configurações da plataforma.
  • guardiões_tenda: Guardiões de tenda, vinculados às tendas e perfis de usuários.
  • tendas: Registros de tendas (grupos), vinculados às cidades, estados, etapas e paróquias.
  • perfis_usuários: Perfis de usuários, vinculados às tendas e outros perfis (cônjuge).
  • funções_serviço_usuários: Mapeamento entre usuários e funções de serviço.

Detalhamento do Banco de Dados

Esta seção apresenta uma visão detalhada da estrutura do banco de dados, incluindo tabelas, políticas de segurança (RLS), papéis de usuário e permissões. O objetivo é explicar de forma clara como os dados são organizados, protegidos e acessados na plataforma.

Organização dos Schemas

  • auth: Gerenciamento de autenticação e usuários.
  • public: Dados principais da aplicação.
  • storage: Gerenciamento de arquivos e imagens.
  • realtime: Mensageria em tempo real.
  • vault: Gerenciamento de segredos e informações sensíveis.

Principais Tabelas e Relacionamentos

O banco de dados possui diversas tabelas, cada uma com suas chaves primárias, estrangeiras e políticas de acesso. Exemplos:

  • cidades: Relacionada a estados (cada cidade pertence a um estado).
  • famílias: Relacionada a tendas (cada família pertence a uma tenda).
  • missas: Relacionada a paróquias.
  • paróquias: Pode ser relacionada a cidades e estados.
  • padres: Relacionado a perfis de usuário e paróquias.
  • tendas: Relacionada a cidades, estados, etapas e paróquias.
  • perfis de usuário: Relacionados a tendas e outros perfis (cônjuge).

Políticas de Segurança (RLS)

Todas as tabelas possuem Row Level Security (RLS) ativado. Isso significa que o acesso aos dados é controlado por políticas que determinam quem pode visualizar, inserir, atualizar ou excluir informações, de acordo com o papel do usuário.

  • devAdmin e Founder: Acesso total a todas as tabelas.
  • Guardião: Acesso de leitura à maioria das tabelas, podendo gerenciar dados do seu grupo/tenda.
  • Usuário: Pode visualizar dados relevantes e atualizar seu próprio perfil.
  • Visualizador: Apenas leitura de dados públicos.

Controle de Acesso Baseado em Papéis (RBAC)

O sistema utiliza RBAC (Role-Based Access Control), onde cada usuário recebe um ou mais papéis que determinam suas permissões. Exemplos de papéis:

  • devAdmin: Administrador técnico, acesso total.
  • Founder: Fundador, acesso total.
  • Guardião: Líder de grupo/tenda.
  • Usuário: Usuário autenticado comum.
  • Visualizador: Apenas leitura.
  • admin: Administrador geral, permissões intermediárias.

Funções de Serviço

Além dos papéis principais, existem funções de serviço para gestão de equipes e atividades específicas, como:

  • Líder de Equipe Consagrados
  • Líder de Equipe Tsedaka
  • Líder de Equipe GAA
  • Líder de Equipe Berçário/Creche
  • Líder de Equipe Ventania
  • Treinador de Equipe de Trabalho
  • Padre Paroquial
  • Participante de Equipe

Como as Permissões São Aplicadas

As permissões são aplicadas principalmente via RLS, garantindo que cada usuário só acesse o que lhe é permitido. Funções de serviço permitem uma gestão mais granular, especialmente para líderes de equipe e responsáveis por grupos.

Notas e Recomendações

  • Todas as tabelas do schema auth são gerenciadas pelo Supabase Auth.
  • O schema storage é usado para arquivos e imagens, com políticas específicas para fotos de família.
  • O schema realtime gerencia funcionalidades de mensagens em tempo real.
  • O schema vault é utilizado para informações sensíveis e segredos.
